# Pedagogy Engine S3.2 — Live Between-Turn Coach Chips (design spec)

**Status:** Approved, pre-implementation (2026-06-23). Realizes the **S3.2** row of `PEDAGOGY_ENGINE_S3.md §1` and the §6.2 Conversation-Sidecar / coach-track architecture. Sibling to the approved S3.1 spec (`2026-06-23-pedagogy-s3.1-post-task-coach-review-design.md`).

**Builds on (shipped):** S3.1 post-task coach review (`PEDAGOGY_ENGINE_COACH_REVIEW=1` live) — the pure `pedagogy/coach_review.py` rubric module, the dedicated `database.update_practice_session_analysis_state` write (the S3.1 cache-persist fix), and the `analysis_state` storage contract (invariant 9).

---

## 0. Goal

While an assignment-linked practice session is live, a **parallel, cheaper correction model** analyzes each learner turn and writes a terse finding to a **side channel**, rendered as a compact **Feedback sidecar chip** on both the voice and text surfaces. Silent — it never interrupts the main conversation. Per §6.2: *flow by default, uptake by exception.*

This is the live half of §6.2's Feedback mode. The S3.1 post-task review is the accumulated-queue half; the two coexist in the sidecar's Feedback mode and remain **independent** in S3.2 (no dedup/merge — deferred).

## 0.1 Decisions locked (brainstorm)

| Decision | Choice | Rationale |
|---|---|---|
| **Surface scope** | **Both voice + text** | Mirrors S3.1. Voice carries the split-attention constraint (below). |
| **Chip trigger** | **Heuristic-gated** | Spend a coach-model call only on turns where the existing per-turn heuristic events already flagged a corrective signal. Cheapest at student scale; reuses analytics already firing. The S3.1 post-task pass is the full-transcript safety net for what live heuristics miss. |
| **Chip lifecycle** | **Persisted** in `analysis_state['coach_chips']` | Durable across reload / voice reconnect; the substrate S3.3 (promote-back) and the L7 teacher debrief read later. Writes are rare (gated). No new store (invariant 9). |
| **Transport** | **Dedicated per-turn endpoint** (`POST /api/practice-sessions/<id>/coach-chip`) | The per-turn analog of S3.1's dedicated generate-on-read endpoint. Clean separation from the analytics event POST; latency isolated; the frontend controls *when* it fires (so voice never shows a chip mid-speech); fail-open per call. |
| **Pure-module reuse** | **Thin chip-specific prompt+parser in `coach_review.py`** (not the full-review builder verbatim) | The per-turn decision ("one focus or stay silent," §6.1 step 3) is a genuinely different prompt than "summarize the whole session." Same rubric constants, same `ReviewItem`, same anti-sycophancy/locale rules — a focused per-turn entry point. Approved as more faithful to §6.1 than literal reuse. |

## 1. Architecture — pure / impure split (mirrors S3.1 / S2)

```
PURE   backend/services/pedagogy/coach_review.py   (extend; stdlib only — import-boundary clean)
         + build_coach_chip_prompt(recent_turns, targets, feedback_policy, surface, ui_language) -> [system, user]
             • asks for AT MOST ONE correction for the LATEST learner turn, or explicit null
             • reuses the locale rule (why in ui_language; utterance/better verbatim in target language),
               the anti-sycophancy posture, and (voice) the confidence_caveat rule
         + parse_coach_chip(raw, *, surface, known_targets=None) -> ReviewItem | None
             • reuses ReviewItem + the work_on item validators; returns None when no usable correction
             • raises ValueError only on a non-dict payload (caught by the orchestrator)

IMPURE backend/services/coach_chip_service.py   (new orchestrator — imports OpenAI/db)
         • generate_coach_chip(deps, bootstrap, uid, session_id, turn_index) -> dict | None
         • COACH_CHIP_MODEL = "gpt-5.4-mini-2026-03-17", reasoning_effort="high"

GATE   backend/services/pedagogy/integration.py
         • coach_chips_enabled()   (reads PEDAGOGY_ENGINE_COACH_CHIPS; mirrors coach_review_enabled())

ROUTE  backend/routes/curriculum_admin.py
         • POST /api/practice-sessions/<session_id>/coach-chip   -> thin wrapper over generate_coach_chip

FRONT  frontend/src/api/coachChips.ts                      • postCoachChip(sessionId, turnIndex) -> CoachChip | null
       frontend/src/components/learning/FeedbackSidecar.tsx (new — compact accumulating chip strip)
         • mounts in AssignmentPracticeWorkspace.tsx beside the conversation; fired at the between-turn breakpoint
```

**Why this split (same invariant as S3.1):** `test_pedagogy_engine_s1.ImportBoundaryTestCase` forbids `pedagogy/*.py` core modules from importing OpenAI/Canvas/resolver/compliance. The chip rubric/prompt/schema/validation are **pure** (`coach_review.py`); the transcript-window fetch + OpenAI call + db append + fail-open wrapping live in the **impure** orchestrator (`coach_chip_service.py`). The existing import-boundary test already covers `coach_review.py` and extends to the new pure functions unchanged.

## 2. Data contract — `analysis_state['coach_chips']`

Sits beside S2's `analysis_state['coverage']` and S3.1's `analysis_state['coach_review']`. `default_analysis_state()` / `normalize_analysis_state()` (`practice_analytics.py`) carry a `coach_chips` key, default `[]`.

```jsonc
[
  {
    "turn_index": 7,                       // the learner turn this chip is about
    "generated_at": "<iso8601>",           // stamped by the impure layer
    "model": "gpt-5.4-mini-2026-03-17",    // provenance — MODEL-VERIFIED (vs heuristic events)
    "surface": "voice" | "text",
    "utterance": "Yo va al tienda",        // learner's ACTUAL words, target language
    "better": "Yo voy a la tienda",        // corrected form, target language
    "why": "'ir' is irregular: yo voy.",   // explanation in ui_language
    "target": "focus_grammar:present-irregular" | null,  // ties to an assignment target when applicable
    "confidence_caveat": false             // voice only: true when the correction leans on low-ASR-confidence audio
  }
]
```

**Contract invariants** (inherited from S3.1):
- **Append-only** through the session; ordered by `turn_index`. One chip max per gated-in turn (the model emits one focus or none).
- **Idempotent per turn:** appending a chip for a `turn_index` that already has one is a no-op (re-fires on reconnect/retry don't duplicate).
- **Locale-parametric.** `utterance`/`better` verbatim in the target language; `why` in the learner's `ui_language`. No hard-coded language.
- **Model-verified provenance.** The `model` field marks chips as model-verified, distinct from the heuristic `learning_events` (LIMITATIONS #7/#8) and S2 coverage (#53(g)).
- **Additive.** Writing `coach_chips` never mutates `coverage`, `coach_review`, `learning_events`, or `session_summary`.

## 3. The heuristic gate (where cost is controlled)

A coach-model call is spent **only** when the just-completed turn emitted a corrective heuristic signal — any of:
`feedback.recast`, `feedback.elicitation`, `feedback.review_item` (the main tutor's correction moves, detected from its response), or `metric.repeated_error`.

- No such signal → `generate_coach_chip` returns `None` with **zero LLM cost** (the dominant case — flowing turns produce no chip).
- A gated-in turn still may yield **no chip**: the chip prompt instructs "one focus or stay silent," and `parse_coach_chip` returns `None` when the model declines. So gating bounds *whether we pay*; the model bounds *whether we speak*.

The gate reads the persisted `learning_events` for `session_id` filtered to `turn_index` and checks for any of the corrective event types above. Ordering is guaranteed by the frontend: it `await`s the per-turn event POST (which persists the signals) **before** calling the coach-chip endpoint.

**Division of labor with S3.1:** live chips are cheap, heuristic-triggered, and model-verified *when triggered*; the S3.1 post-task review does the full-transcript model pass that catches whatever the live heuristics missed. A live-heuristic blind spot is therefore not a silent gap — it surfaces post-task.

## 4. Data flow (per turn)

```
Learner completes a turn (text exchange, or realtime learner-turn-complete on voice)
  -> frontend reports the turn event (existing POST .../events; persists heuristic signals)   [awaited]
  -> at the between-turn breakpoint, frontend calls postCoachChip(sessionId, turnIndex)
     -> POST /api/practice-sessions/<id>/coach-chip { turn_index }
        -> generate_coach_chip(deps, bootstrap, uid, session_id, turn_index):
            1. coach_chips_enabled()?  no -> return None                  (zero reads/LLM when flag off)
            2. assignment-linked + engine targets resolvable?  no -> return None
            3. heuristic gate: did turn_index emit a corrective signal?  no -> return None  (no LLM)
            4. already a chip for turn_index in analysis_state['coach_chips']?  yes -> return it (no LLM)
            5. fetch a small recent-turn transcript window (the last ~3 turns, learner + tutor,
               enough context to judge the latest learner turn without re-sending the whole transcript)
            6. msgs = build_coach_chip_prompt(window, targets, feedback_policy, surface, ui_language)  [pure]
            7. client.chat.completions.create(model=COACH_CHIP_MODEL, reasoning_effort='high',
                                              response_format={'type':'json_object'}, messages=msgs)
            8. chip = parse_coach_chip(json.loads(resp), surface=..., known_targets=...)  [pure]
               chip is None -> return None   (model stayed silent)
            9. re-read session; append serialize(chip) to analysis_state['coach_chips']
               via deps.db.update_practice_session_analysis_state(session_id, state, sql_engine=deps.sql_engine)
           10. return the serialized chip
            (steps 2-9 wrapped in try/except -> return None; logger.exception; never raises)
        -> { success: True, coachChip: <obj> | null }
  -> sidecar appends the chip and renders it at the breakpoint (voice: only when not speaking)
```

**The conversation never waits on this.** The frontend fires-and-renders asynchronously; a slow or failed chip request never blocks the next turn or the main tutor.

## 5. Voice constraint (split-attention)

The frontend fires `postCoachChip` at the **between-turn breakpoint** and renders chips **only when the learner is not actively speaking**:
- **Text:** the breakpoint is after the exchange completes (trivial).
- **Voice:** the breakpoint is the realtime **learner-turn-complete** signal; the chip renders silently in the sidecar and never appears mid-speech. (The exact `useRealtimeChat` turn-lifecycle hook is a plan-level detail; the realtime API exposes turn events.)

Chips are terse one-liners (the corrected form + a short why), expand-on-tap. Legal windows only: silent write / between-turn chip / post-task review — never require reading while speaking.

## 6. Fail-open invariants

Every failure path resolves to `coachChip: null` — never a 500, never a blocked turn: flag off · not assignment-linked · no engine targets · no heuristic gate signal · no transcript window · OpenAI error/timeout/rate-limit · malformed model JSON (`parse_coach_chip` raises → caught) · 0-row analysis_state write (legacy Firestore-only session → warn, not raise — inherited from the S3.1 fix). The sidecar always has a graceful empty state (no chips yet).

When the flag is off the route gate skips bootstrap and the service gate precedes its reads/LLM, so a flag-off request runs only the cheap session-ownership lookup.

## 7. Scope boundary (non-goals — deferred to later slices)

- **Promote-back into the main conversation** (the main tutor surfacing a repeated/hard-target error for in-the-moment self-repair) → **S3.3**. In S3.2 chips are silent and side-channel only.
- **Main tutor goes correction-light** → **S3.3**. In S3.2 the **main tutor is unchanged** — it still corrects inline; chips are *additive and silent* (they may echo the tutor's correction as a persistent structured record, which is the substrate S3.3 uses when the tutor goes quiet).
- **Ask mode** (learner-initiated quick help) → **S3.4**.
- **Chip ↔ post-task-review dedup/merge** — independent in S3.2; deferred.
- **Raw-tutor / free-chat sessions** — no engine targets → no chips (same gate as S3.1 / S2).
- **Cost accounting** folded into `cost_summary` — only provenance fields recorded.

## 8. Flag & rollout — strangler-fig (same cadence as S1/S2/S3.1)

New flag **`PEDAGOGY_ENGINE_COACH_CHIPS`** (default off), independent of `PEDAGOGY_ENGINE_COACH_REVIEW` and `PEDAGOGY_ENGINE_RECYCLING`:
1. Land code behind the flag, default `'0'`; wire into `cloudbuild.yaml` `--set-env-vars` + `substitutions` (`_PEDAGOGY_ENGINE_COACH_CHIPS: '0'`). REPLACE semantics: the var MUST be listed AND its substitution default MUST match the live value (currently ABSENT → off, so `'0'` is REPLACE-safe).
2. Deploy inert (`'0'`) → verify health.
3. Flip live: `--update-env-vars PEDAGOGY_ENGINE_COACH_CHIPS=1`.
4. Burn-in (live-verify both surfaces: a flagged turn produces a chip in the sidecar; a flowing turn does not; flag-off → no chips) → bump cloudbuild default `'0'→'1'`.
5. Rollback instant: `--update-env-vars PEDAGOGY_ENGINE_COACH_CHIPS=0`.

**Persistence reminder (S3.1 lesson):** the per-turn append uses `update_practice_session_analysis_state`, NOT `update_practice_session` (the latter self-disables under the live `DUAL_WRITE_ANALYTICS_EVENTS=1`).

## 9. Testing & eval

**Deterministic units (gate `make test-backend`):**
- Pure (`test_pedagogy_engine_s3.py` or a sibling): `build_coach_chip_prompt` (one-focus instruction present, voice caveat rule, `ui_language` threaded, targets present); `parse_coach_chip` (returns `ReviewItem` for a valid one-item payload; `None` for null/empty/no-utterance; non-dict raises; `target` normalized against known targets; `confidence_caveat` honored on voice / forced false on text). Extended `ImportBoundaryTestCase` (the new pure functions add no forbidden imports).
- Impure (`coach_chip_service`, fake db/client à la S3.1's `_FakeDb`/`_FakeDeps`): flag-off → zero reads/LLM; no heuristic signal → zero LLM; fail-open (raising db/client → `None`); model-returns-null → `None`, no append; turn-dedup (existing chip for `turn_index` → returned, no 2nd LLM call); append writes via `update_practice_session_analysis_state`.
- Route (`POST .../coach-chip`): 404 missing / 403 not-owner / flag-off → `coachChip: null` with no bootstrap; gated path returns chip-or-null.

**Frontend (vitest):** `FeedbackSidecar` renders an accumulating chip list + empty state; fires `postCoachChip` at the breakpoint; never renders a chip while speaking (voice state); the `postCoachChip` wrapper.

**Behavioral eval (opt-in `RUN_PEDAGOGY_EVAL=1`, NOT CI):** per-turn seeded-error slices → real `gpt-5.4-mini-2026-03-17` → LLM-judge on chip **false-correction rate** and one-focus discipline (never more than one correction). Reuses the S2/S3.1 eval scaffolding + the deterministic `_coerce_judge_verdict`; the eval model is **pinned** (no env override).

**Live verify (playwright):** both surfaces — drive a session, hit a turn that trips a heuristic signal → confirm a chip appears in the sidecar and persists (survives reload); a flowing turn → no chip; flag-off → no chips.

## 10. Open questions / future hooks

- **S3.3 reuse:** `coach_chips` is the durable substrate for promote-back — S3.3 reads repeated/hard-target chips and surfaces them into the main channel, then takes the main tutor correction-light. The `feedbackPolicy.mode` + `elicitation_repeat_threshold` become the promote-back policy (no new config, §6.2).
- **L7 teacher debrief:** `coach_chips` (model-verified, structured) feed the after-conversation teacher surface later.
- **Chip ↔ post-task-review dedup:** once both are live, the post-task review could dedup against accumulated chips. Deferred; independent in S3.2.
