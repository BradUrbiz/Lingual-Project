# Pedagogy Engine — S3.3: Promote-Back + Correction-Light Main Tutor (design)

**Status:** Design (approved, pre-implementation). Sibling to the S3.1 / S3.2 design specs. Realizes the **S3.3 row** of `PEDAGOGY_ENGINE_S3.md` §1 and the §6.2 coach-track "promote-back" architecture in `PEDAGOGY_ENGINE.md`.

**Depends on:** S3.2 (live coach chips) — **CUT OVER 2026-06-24** (`PEDAGOGY_ENGINE_COACH_CHIPS=1`). Promote-back is layered on the S3.2 per-turn chip round-trip; it adds no new endpoint and no second LLM call.

---

## 0. TL;DR

S3.2 surfaces silent between-turn coach chips into a sidecar; the main tutor is unchanged and never sees them. **S3.3 closes the loop:** when an error *repeats* (crosses the teacher's `elicitation_repeat_threshold`) or lands on a teacher **hard-target** (`focus_grammar`), the coach **promotes it back into the main conversation** — the tutor invites in-the-moment self-repair *in its own words* ("Earlier you said X — want to try that again?"). In exchange, the main tutor goes **correction-light**: a separate coach owns error correction, so the tutor's job collapses to *hold a good conversation + voice the occasional coach note*. This is the structural mitigation for the ~30% voice instruction-adherence ceiling (it **subtracts** instruction load rather than adding it).

The decision of *what to promote* is **deterministic, pure, and unit-testable** (a recurrence counter + mode-modulated thresholds + three over-promotion guards). The coach LLM call is the same single-turn call S3.2 already makes. Injection reuses an **already-in-production** pattern (the avatar-context system-message injection on voice). One new independent flag: `PEDAGOGY_ENGINE_PROMOTE_BACK` (default off).

---

## 1. Goal & scope

**Goal.** When the coach detects a repeated or hard-target error during a live assignment practice session, surface it into the *main* conversation for in-the-moment self-repair, while the main tutor runs correction-light. Flow by default, uptake by exception.

**In scope:**
- A **pure decision layer** (`pedagogy/promote_back.py`) deciding promote/no-promote from deterministic signals.
- **In-character injection** of the promote-back into the main channel on **both** surfaces (voice + text).
- A flag-gated **correction-light** main-tutor stance (session-start prompt change, both surfaces).
- Additive `analysis_state` storage (`promote_back_state` counter + `promotions[]` log) — invariant 9, no new store.
- Flag-gated rollout (`PEDAGOGY_ENGINE_PROMOTE_BACK`, default off), deterministic units, opt-in eval, fail-open throughout.

**Non-goals (deferred):**
- **Ask mode** (learner-initiated quick help) → S3.4.
- **Cross-model intent dedup** (detecting that the correction-light tutor happened to address an error anyway) — bounded structurally instead (see §4.5); the post-task review is the backstop.
- **A second coach LLM call** for the promote decision — the decision is deterministic post-processing on the chip the S3.2 call already produced.
- **L7 / S2 cross-consumption** of `promotions[]` — the log is structured for that later, but S3.3 builds only the live learner-facing behavior.
- **Raw-tutor / custom-prompt / free-chat sessions** — engine off; `is_custom_prompt` short-circuits the render (no correction-light), and with no chips there are no promotions.

## 2. Architecture — pure / impure split (mirrors S2 / S3.1 / S3.2)

```
PURE   backend/services/pedagogy/promote_back.py            (NEW — stdlib only, import-boundary clean)
         • PromoteDecision dataclass {promote: bool, signature: str, reason: str}   (no rendered text — pure)
         • error_signature(chip) -> str
         • mode_threshold(feedback_policy) -> (regular_threshold, hard_target_threshold)
         • decide_promote_back(promote_state, chip, feedback_policy, turn_index)
                                                  -> (PromoteDecision, updated_state)
         • build_promote_prompt(chip, surface) -> str   (called by the orchestrator when promote==True; the coach note is a tutor-facing instruction in the working language)

IMPURE backend/services/coach_chip_service.py               (MODIFY generate_coach_chip)
         • after the chip is built, when promote_back_enabled():
             decision, new_state = decide_promote_back(...); if decision.promote:
               promote_prompt = build_promote_prompt(chip, surface)   (surface already resolved here)
             persist new_state (promote_back_state) + append promotions[] into the SAME analysis_state write,
             fold promote/promote_prompt/promote_reason into the chip

GATE   backend/services/pedagogy/integration.py             (MODIFY)
         • promote_back_enabled()  (reads PEDAGOGY_ENGINE_PROMOTE_BACK; mirrors coach_chips_enabled())
         • resolve_assignment_system_prompt computes
             correction_light = promote_back_enabled() and coach_chips_enabled()
           and passes it to render_assignment_prompt(plan, surface, correction_light=...)

RENDER backend/services/pedagogy/render/assignment_prompt.py (MODIFY — correction_light param)
       backend/services/assignment_resolver.py               (MODIFY _build_tutor_stance — correction_light variant)

ROUTE  backend/routes/curriculum_admin.py                    POST /coach-chip  (NO new endpoint — enriched chip flows through)

FRONT  frontend/src/api/coachChips.ts                        CoachChip gains promote?/promote_prompt?/promote_reason?
       frontend/src/hooks/useRealtimeChat.ts                 injectPromoteBack(prompt) — reuses avatar-context queue+flush
       frontend/src/components/assignments/AssignmentPracticeWorkspace.tsx
         • triggerCoachChip: on a LIVE chip with promote===true → inject in-character
           (voice: avatar-pattern system message + response.create at breakpoint;
            text: hold a pending coach-note, attach to the next POST /messages turn)
```

**Why this split.** The import-boundary invariant (`test_pedagogy_engine_s1.ImportBoundaryTestCase`) forbids `pedagogy/*` core modules from importing OpenAI / Canvas / resolver / compliance. The **entire promote decision is pure** (`promote_back.py`); the impure orchestrator (`coach_chip_service.py`) does only I/O and persistence. This is the same discipline that kept `coverage.py` (S2) and `coach_review.py` (S3.1/S3.2) pure.

**Two-flag safety invariant.** Correction-light activates only when `promote_back_enabled() AND coach_chips_enabled()`. This closes the dangerous gap: the main tutor never stops correcting unless the coach can actually promote. (promote-on / chips-off ⇒ tutor stays normal, no under-correction; this is also why the render seam reads *both* flags.)

## 3. Data contract — `analysis_state` (additive, invariant 9 — no new store)

Two new keys beside `coverage` (S2), `coach_review` (S3.1), `coach_chips` (S3.2). `default_analysis_state()` / `normalize_analysis_state()` (`practice_analytics.py`) carry them.

```jsonc
"promote_back_state": {                 // deterministic counter + guardrail bookkeeping (pure-module owned)
  "counts": { "focus_grammar:present-irregular": 3 },   // per-signature recurrence count
  "last_promoted_turn": 12,             // cooldown enforcement
  "promoted_count": 2                   // per-session cap enforcement
}

"promotions": [                         // durable log of what was promoted (analytics/L7 + record; NEVER re-injected)
  {
    "turn_index": 12,
    "signature": "focus_grammar:present-irregular",
    "reason": "repeat" | "hard_target",
    "prompt": "<in-character self-repair instruction>",
    "generated_at": "<iso8601>"
  }
]
```

The existing **`coach_chips`** entry is enriched (promote fields ride the chip the service already returns):

```jsonc
{ ...existing S3.2 chip fields...,
  "promote": true,
  "promote_prompt": "<in-character self-repair instruction>",
  "promote_reason": "repeat" | "hard_target" }
```

**Contract invariants:**
- **Additive.** Writing `promote_back_state` / `promotions` / the promote fields never mutates `coverage`, `coach_review`, other chips, `learning_events`, or `session_summary`.
- **Hydration safety.** `GET /coach-chips` keeps rendering chips *silently* (existing behavior). Injection fires only in the live `triggerCoachChip` path on a freshly-returned `promote:true` chip — **never** on hydration/reload — so reopening a session never re-fires the tutor. `promotions[]` is the durable record.
- **Locale-parametric.** Quoted `utterance`/`better` in the target language; instruction/explanation text in the working/`ui_language`. No hard-coded language.

## 4. Decision policy & guardrails (pure `promote_back.py`)

All deterministic, no LLM. Fully unit-testable.

### 4.1 Candidacy
A promote candidate exists **only** on a turn that already produced a chip (the S3.2 heuristic gate fired → a corrective signal was present). Promote-back is a strict subset of chip turns; no chip → no promote; the coach LLM is never called a second time.

### 4.2 Error signature — `error_signature(chip)`
- If `chip.target` is set → use it verbatim (e.g. `focus_grammar:present-irregular`) — the most stable, teacher-aligned key.
- Else → normalized `chip.better` (the corrected form): lowercase, accent-stripped via stdlib `unicodedata` (NFKD → ASCII), whitespace-collapsed.

### 4.3 Threshold — `mode_threshold(feedback_policy)` (reuses the teacher knob)
Base = `feedback_policy.elicitation_repeat_threshold` (default **2** when unset; the implementer confirms the exact field name against the policy normalizer in `pedagogy/policies.py` / `assignment_resolver` — the default-when-unset fallback keeps the code correct even if the knob is absent from a given policy dict). Effective thresholds by `mode` (§6.2: "fluency rarely / accuracy sooner"; hard-targets promote one repetition sooner — §9 "rule-based focus_grammar → elicit first"):

| `mode` | regular-error threshold | hard-target threshold |
|---|---|---|
| `accuracy_first` | `max(1, base − 1)` | `max(1, base − 2)` |
| `balanced` | `base` | `max(1, base − 1)` |
| `fluency_first` | `base + 1` | `base` |

### 4.4 `decide_promote_back(promote_state, chip, feedback_policy, turn_index) -> (PromoteDecision, updated_state)`
Pure; returns a **new** state (never mutates the input). Does **not** build the rendered prompt — that is the orchestrator's job (§2), keeping the decision free of surface/ui_language concerns. Steps:
1. `sig = error_signature(chip)`; `counts[sig] += 1`.
2. `reason = "hard_target" if sig.startswith("focus_grammar:") else "repeat"`; pick the matching threshold from §4.3.
3. **Guardrails — all must pass to promote:**
   - **Threshold:** `counts[sig] >= threshold`.
   - **Cooldown:** `turn_index - last_promoted_turn >= PROMOTE_COOLDOWN_TURNS` (default **2**).
   - **Per-session cap:** `promoted_count < max_promotions_per_session` — by mode: fluency **2** / balanced **3** / accuracy **5**.
4. If promoting: set `last_promoted_turn = turn_index`, `promoted_count += 1`, **`counts[sig] = 0`** (reset-on-promote). Return `PromoteDecision(promote=True, signature=sig, reason=reason)`. The orchestrator then calls `build_promote_prompt(chip, surface)`.
5. Else: return `PromoteDecision(promote=False, signature=sig, reason=reason)` with the updated `counts`.

**Tunable constants** (module-level, named): `DEFAULT_REPEAT_THRESHOLD = 2`, `PROMOTE_COOLDOWN_TURNS = 2`, `MAX_PROMOTIONS_BY_MODE = {"fluency_first": 2, "balanced": 3, "accuracy_first": 5}`.

### 4.5 Three orthogonal over-promotion guards
- **Cooldown** → no back-to-back interruptions (turn-adjacency).
- **Per-session cap** → no correction gauntlet (session-level).
- **Reset-on-promote** → a still-unfixed recurring error is not promoted every subsequent turn; it must re-accumulate a full threshold (same-error level). *This is the load-bearing anti-nag rule:* without it, "uptake by exception" degrades into nagging.

### 4.6 Two-model dedup
Correction-light instructs the main tutor **not** to correct, so the coach structurally owns correction; double-correction risk is low by construction. Residual risk (a correction-light tutor occasionally addressing something anyway) is bounded by the three guards above. We deliberately do **not** build cross-model intent-matching (the §6.2 risk note flags it as hard); the S3.1 post-task review backstops anything missed.

### 4.7 `build_promote_prompt(chip, surface) -> str`
Pure. Produces the **coach note** handed to the tutor (the tutor phrases the self-repair in its own words — the chosen "in-character" voicing). The note is a tutor-facing *instruction* in the prompt's working language; the quoted `utterance`/`better` stay verbatim in the target language (so no `ui_language` param is needed). Routing follows §9:
- rule-based `focus_grammar` (reason `hard_target`) → **elicitation** ("Earlier the learner said '<utterance>' — briefly invite them to try '<better>' again, then continue.").
- lexical/expression slip (reason `repeat`, non-grammar) → **recast-style** ("Briefly model '<better>' for '<utterance>' and move on.").
- Terser on `surface == "voice"`.

## 5. Injection (frontend) — in-character, both surfaces

On a **live** chip with `promote === true`, `triggerCoachChip` still appends to the sidecar (additive, as today) **and** routes the coach note to the main channel:

### 5.1 Voice — reuse the proven avatar-context path
`injectPromoteBack(promote_prompt)` pushes the prompt as a queued `system` conversation item; it flushes at the between-turn breakpoint under the **same** `shouldTriggerAvatarContextResponse` guard (`isConnected && !isListening && !isSpeaking && no active response`) → `createConversationItem({type:'message', role:'system', content:[{type:'input_text', text: promote_prompt}]})` + `createRealtimeResponseUnlessHeld()`. The realtime tutor then voices the self-repair in its own words.
- **Latency reality:** the coach LLM takes ~1–3s, so the chip can return *after* this turn's `response.done` already flushed. We push onto the queue **and** attempt an immediate guarded flush; if the learner is already mid-speech, the note stays queued and flushes at the **next** breakpoint. Net: promote-back may land one turn late on voice — acceptable for "uptake by exception," and it never interrupts live speech.
- Reuses (or mirrors) `queuedAvatarContextsRef` + `createConversationItem` + `createRealtimeResponseUnlessHeld` (useRealtimeChat.ts ~693–724) — an in-production injection pattern.

### 5.2 Text — ride the next turn's prompt
Text has no live channel (REST round-trip per turn). The pending `promote_prompt` is held frontend-side (e.g. `pendingPromoteBackRef`, keyed by session) and attached to the learner's **next** `POST /messages` turn as a transient coach note prepended to that turn's system prompt; the text model (which adheres reliably) weaves the self-repair into its reply in its own words, then the note is cleared. Rides the existing turn — no extra round-trip — and stays consistent with "tutor's own words." (An unprompted tutor message with no learner turn would be odd in a text thread.)

### 5.3 Hydration
Unchanged — `GET /coach-chips` renders silently; injection only ever fires on a fresh live `promote:true`.

## 6. Correction-light prompt (the high-scrutiny change)

`resolve_assignment_system_prompt` → `render_assignment_prompt(plan, surface, correction_light)` → `_build_tutor_stance(..., correction_light)`. When `correction_light` is True the stance:
- tells the tutor a **separate coach owns error correction** — prioritize a natural, flowing conversation; **do not interrupt to correct** grammar/vocabulary slips;
- **stays feedback-aware** (correction-light ≠ feedback-blind): still clarify on communication breakdown / low understanding (meaning-level, invariant 4); still give brief confirmative acknowledgment on success (anti-sycophancy);
- **expects between-turn coach notes** and delivers them naturally in its own words (primes the model to voice the injected promote-backs).

This **replaces** the normal target-type-aware correction-routing block (the prompt-first/recast ladder built from `plan.targets`) — under correction-light that routing is the coach's job. Voice keeps critical-rules-last ordering. Fully flag-gated: `correction_light=False` → **byte-identical** to today's prompt on both surfaces (verified by a golden-prompt test).

**Architectural payoff (§6.2).** Correction-light *subtracts* instruction load (drop the whole correction ladder) instead of adding to it. The ~30% voice-adherence ceiling worsens as instructions stack, so the structural fix makes the tutor's job *smaller* — "hold a good conversation" + "voice the occasional coach note" — and moves accuracy-tuned correction to a model that isn't fighting flow constraints. Promote-back injection is the seam that reconnects the two.

## 7. Fail-open invariants (§0.1) — same posture as S3.2

Promote-back can never break or interrupt a conversation:
- **Backend** — the promote decision runs *inside* `generate_coach_chip`'s existing `try/except → None`. If `decide_promote_back` raises, the chip is still returned **without** promote fields (degrade to a silent chip). The state write rides the same `update_practice_session_analysis_state` (re-read-before-write; fail-closed 2000ms; 0-row → warn, not raise). A promote-back failure never costs the chip.
- **Frontend** — `injectPromoteBack` sits under the existing `triggerCoachChip` fail-open catch; injection failure (closed data channel, etc.) still shows the chip in the sidecar and never blocks the session. Text: a failed pending-note attach lets the next turn proceed normally.
- **Correction-light** — flag read / render failure defaults `correction_light=False` → normal prompt. Off → identical to today.
- **Net failure modes** (flag off · no chip · decision raises · injection fails · 0-row state write) all degrade to: *no promote, silent chip, normal conversation.* Never a 500, never a blocked / interrupted / double-injected turn.

## 8. Testing & eval

**Deterministic units (gate `make test-backend`)** — bulk is the pure module:
- `promote_back.py`: `error_signature` (target vs `better` fallback; accent/whitespace normalize); `mode_threshold` (3 modes × regular/hard-target; base default + teacher override; clamps); `decide_promote_back` (counter increment; threshold crossing; hard-target-sooner; **cooldown**; **per-session cap**; **reset-on-promote** — same signature must re-accumulate; no-chip → no candidate; returns new state without mutating input).
- `build_promote_prompt` (elicitation vs recast by reason/target-type; voice terseness; `ui_language` threading; target-language quoting).
- `coach_chip_service` (fake db/client à la S3.2): flag-off → chip unchanged & no `promote_back_state` write; threshold-not-met → chip sans promote; threshold-met → promote fields + `promotions[]` appended + state persisted; fail-open (decide raises → chip still returned).
- **Correction-light golden-prompt test:** `correction_light=False` → byte-identical to current render on both surfaces; `True` → correction ladder dropped, flow + coach-note instructions present, voice critical-rules-last preserved.
- **Extended `ImportBoundaryTestCase`:** `promote_back.py` imports no OpenAI/Canvas/resolver/compliance.

**Frontend (vitest):** `triggerCoachChip` `promote:true` (live) → injects + still appends chip; `promote:false` → no injection; hydration path → never injects; text pending-note attach + clear; `useRealtimeChat.injectPromoteBack` queues + flushes under the guard (mock data channel), stays queued while the learner is speaking.

**Behavioral eval (opt-in `RUN_PEDAGOGY_EVAL=1`, NOT CI):** seeded repeated-error transcripts → promote fires at the right recurrence per mode, honors cap/cooldown (no over-promotion), `build_promote_prompt` output is in-character + correctly routed. Reuses the S2/S3.1 eval scaffolding.

## 9. Flag & rollout — strangler-fig (same cadence as S1/S2/S3.1/S3.2)

New flag **`PEDAGOGY_ENGINE_PROMOTE_BACK`** (default off), independent of the other pedagogy flags:
1. Land code behind the flag, default `'0'`; wire into `cloudbuild.yaml` `--set-env-vars` + `_PEDAGOGY_ENGINE_PROMOTE_BACK: '0'` substitution. **REPLACE semantics:** the deploy uses `--set-env-vars` (replaces the whole env set), so the var MUST be listed AND its substitution default MUST match the live value — currently **ABSENT (off)**, so default `'0'` is REPLACE-safe. Verify every other substitution still matches live before the build.
2. Deploy inert (flag `'0'`) → verify health (prompt byte-identical, no behavior change).
3. Flip live: `gcloud run services update lingual-app --project=lingu-480600 --region us-central1 --update-env-vars PEDAGOGY_ENGINE_PROMOTE_BACK=1`.
4. Burn-in → bump cloudbuild default `'0'→'1'` for durability.
5. **Rollback:** instant via `--update-env-vars PEDAGOGY_ENGINE_PROMOTE_BACK=0` → the tutor reverts to normal correction; no injection.

**Burn-in plan (honest about the voice constraint):** drive a real **text** session on a scaffolded assignment, repeat a target error past threshold → confirm (a) the main tutor runs correction-light (does not pre-empt the coach) and (b) the promote-back is woven into the tutor's next reply in-thread; confirm flag-off / threshold-not-met → no promotion. The **voice** surface is limited by the same headless-WebRTC-mic constraint that capped S3.1/S3.2 burn-in (cannot supply live mic audio); voice shares the surface-agnostic injection path and is covered by units + the in-production avatar pattern — documented as a known burn-in limitation, not a code gap.

## 10. As-built narrowing (to record in LIMITATIONS #53 after cutover)
- Promote decision is **heuristic/deterministic** (counter + thresholds), bounded by the same heuristic blind spots as S3.2's chip gate — locale-bounded feedback catalogs (en/fr/es native; ko/ru/he/tl fall back to generic-English). The post-task review remains the full-transcript safety net.
- **No cross-model intent dedup** — correction-light + the three guards bound double-correction structurally; not eliminated.
- **Voice promote-back may land one turn late** (coach-LLM latency vs. the between-turn breakpoint).
- **Voice burn-in limited** by the WebRTC-mic constraint (shared with S3.1/S3.2).
- `promotions[]` is **not yet** consumed by S2 recycling or the L7 teacher debrief (structured for it; not wired).
