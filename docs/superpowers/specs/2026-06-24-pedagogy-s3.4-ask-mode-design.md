# Pedagogy Engine — S3.4: Ask Mode (learner-initiated quick help) (design)

**Status:** Design (controller-approved under the autonomous build mandate — no user-review gate; rationale documented inline). Sibling to the S3.1/S3.2/S3.3 design specs. Realizes the **S3.4 row** of `PEDAGOGY_ENGINE_S3.md` §1 and the **Ask** half of `PEDAGOGY_ENGINE.md` §6.2's Conversation Sidecar.

**Independence:** S3.4 is **largely independent of the correction track** (S3.1/3.2/3.3). It adds a parallel learner-initiated quick-help channel; it does not read or write `coach_chips` / `coach_review` / `promote_back_state`, and shares no flag with them.

---

## 0. TL;DR

The Conversation Sidecar gets a second mode: **Ask**. The learner types a short question — "how do I say X?", "what does Y mean?", "give me a hint", "translate this" — and receives a **terse, scaffold-not-answer** reply that returns responsibility to the learner. The defining constraint is **anti-answer-dump**: Ask must never produce the student's full target-task answer (it offers a hint / forced choice / short model + retry, or politely redirects), and Ask exchanges are logged to a **separate `ask_log`, never `learning_events`** — so analytics never count scaffolded help as student production (§6.2; §15 "Ask-mode answer outsourcing" risk). Pure/impure split mirrors the coach modules. New flag `PEDAGOGY_ENGINE_ASK_MODE` (default off), fail-open throughout.

---

## 1. Goal & scope

**Goal.** A learner-initiated quick-help channel in the sidecar, available on both surfaces, that gives brief scaffolded help inside the assignment's scope without doing the learner's work or polluting their production evidence.

**In scope:**
- A `POST /api/practice-sessions/<id>/ask` endpoint: takes a learner question, returns one terse scaffolded answer.
- A pure `pedagogy/ask.py` (prompt builder + parser + serializer + the anti-answer-dump contract).
- An impure `ask_service.py` orchestrator (flag-gated, fail-open, logs to `analysis_state['ask_log']`).
- A `ConversationSidecar` wrapper with a **Feedback | Ask** toggle; a new `AskPanel` for the Ask mode (the existing `FeedbackSidecar` is reused unchanged for Feedback mode).
- One `askModeEnabled` field on the launch bootstrap so the Ask tab is hidden while the flag is off.
- Flag-gated rollout (`PEDAGOGY_ENGINE_ASK_MODE`, default off), deterministic units, opt-in eval, fail-open.

**Non-goals (deferred / YAGNI):**
- **Replay + pronunciation comparison** — §6.2 lists these as "deferred extras." Not in S3.4.
- **Voice-channel injection** — Ask never speaks into the realtime session; it's side help (text in/out). This keeps it independent of the correction track and free of split-attention concerns.
- **Ask answers as evidence** — explicitly excluded from `learning_events` and from S2 coverage / S3 analysis. Ask is help, not production.
- **Multi-turn Ask threads / memory** — each Ask is a single Q→A with a small read-only transcript window for context; no conversational Ask state beyond the logged list.
- **Teacher-visible Ask analytics** — `ask_log` is structured for a future L7 "help usage" surface, but S3.4 builds only the learner-facing channel.

## 2. Architecture — pure / impure split (mirrors S3.1/3.2/3.3)

```
PURE   backend/services/pedagogy/ask.py                     (NEW — stdlib only, import-boundary clean)
         • AskAnswer dataclass {answer: str, kind: str}      (kind ∈ hint|translation|definition|clarification|phrase|refusal)
         • build_ask_prompt(question, recent_turns, targets, feedback_policy, scaffold_policy, surface, ui_language) -> list[msg]
         • parse_ask_answer(raw, *, surface) -> AskAnswer | None    (validate/coerce/cap; raises on non-dict)
         • serialize_ask_answer(answer) -> dict

IMPURE backend/services/ask_service.py                       (NEW — imports OpenAI/db)
         • answer_ask(deps, bootstrap, uid, session_id, question, turn_index=None) -> dict | None
         • ASK_MODEL = "gpt-5.4-mini-2026-03-17", reasoning_effort="high"   (project text-LLM convention)

GATE   backend/services/pedagogy/integration.py
         • ask_mode_enabled()  (reads PEDAGOGY_ENGINE_ASK_MODE; mirrors coach_chips_enabled())

ROUTE  backend/routes/curriculum_admin.py
         • POST /api/practice-sessions/<session_id>/ask   ->  thin wrapper over answer_ask (owner+flag gated, fail-open)

BOOTSTRAP  the launch/bootstrap payload gains  askModeEnabled: ask_mode_enabled()   (so the frontend hides the Ask tab while off)

FRONT  frontend/src/api/ask.ts                              • postAsk(sessionId, question, turnIndex?) -> AskAnswer | null
       frontend/src/components/learning/AskPanel.tsx        (new — Ask input + Q&A list + "scaffold, not graded" note)
       frontend/src/components/learning/ConversationSidecar.tsx  (new wrapper — Feedback | Ask toggle; renders FeedbackSidecar or AskPanel)
       AssignmentPracticeWorkspace.tsx                      (swap the FeedbackSidecar mount for ConversationSidecar; pass sessionId + askModeEnabled)
```

**Why this split.** Same import-boundary invariant as the coach modules: the rubric/prompt/parser/anti-answer-dump contract are PURE (`ask.py`, stdlib only, added to `ImportBoundaryTestCase`); the transcript fetch + OpenAI call + `analysis_state` log live in the impure orchestrator. The existing `FeedbackSidecar` is reused by composition (untouched), so S3.2's hydration wiring is not disturbed.

## 3. Data contract — `analysis_state['ask_log']` (additive, invariant 9 — no new store)

A new key beside `coverage`/`coach_review`/`coach_chips`/`promote_back_state`/`promotions`. `default_analysis_state()`/`normalize_analysis_state()` carry it (default `[]`, list-only).

```jsonc
"ask_log": [                            // learner help usage — logged SEPARATELY from student production
  {
    "question": "<learner's question, verbatim>",
    "answer": "<terse scaffolded answer>",
    "kind": "hint" | "translation" | "definition" | "clarification" | "phrase" | "refusal",
    "turn_index": 7 | null,             // the conversation turn the learner was on when they asked (context only)
    "generated_at": "<iso8601>",
    "model": "gpt-5.4-mini-2026-03-17"
  }
]
```

**Contract invariants:**
- **Help ≠ evidence.** `ask_log` is NEVER mirrored into `learning_events`, S2 `coverage`, or S3 analysis. Ask answers do not count as student production (§6.2; §15 mitigation).
- **Additive.** Writing `ask_log` never mutates other `analysis_state` keys.
- **Locale-parametric.** The `answer` explanation renders in the learner's `ui_language`; any quoted target-language form is verbatim target-language. No hard-coded language.
- Persisted via `update_practice_session_analysis_state(...)` (the S3.1 cache-persist path) — NOT `update_practice_session`.

## 4. The anti-answer-dump contract (the heart of S3.4)

`build_ask_prompt`'s system message encodes (verbatim-ish, locale-parametric):
- **Role:** a concise language helper for a learner mid-practice on a specific assignment.
- **Scope:** stay inside the assignment/course scope (the resolved targets are provided); decline off-topic or non-language requests with a brief redirect.
- **Never do the learner's work:** do NOT produce the full answer to the assignment task or complete the learner's current sentence/turn for them. If asked to "just give the answer," respond with a hint, a forced choice, or a short model + an invitation to try — set `kind: "refusal"` for an explicit answer-dump request that you redirect.
- **Brevity:** ≤ 2 sentences (terser on voice). Return responsibility to the learner ("try saying…", "what do you think comes next?").
- **Policy-aware:** respect `feedbackPolicy.mode` (fluency_first → lighter/hint-first; accuracy_first → can be a bit more explicit) and `scaffold_policy` (honor the hint-ladder / max-modeling posture).
- **Output:** strict JSON `{"answer": "...", "kind": "<one of the kinds>"}`. Explanation in `ui_language`; quoted target forms verbatim in the target language.

`parse_ask_answer` validates types, coerces/caps the answer length, normalizes `kind` to the allowed set (unknown → `"clarification"`), and returns `None` for an empty answer (orchestrator → graceful null). Raises `ValueError` only on a non-dict payload (caught → null).

**Why a `refusal` kind:** it makes "the learner tried to get the answer dumped, and we redirected" an explicit, loggable signal — useful later for L7 help-usage analytics and for evaluating the anti-dump guardrail's fire rate.

## 5. Data flow

```
Learner opens the sidecar → Ask tab (shown only when askModeEnabled) → types a question → postAsk(sessionId, question, turnIndex?)
  -> POST /api/practice-sessions/<id>/ask  (owner-gated 404/403 first; route flag gate before bootstrap)
     -> answer_ask(deps, bootstrap, uid, session_id, question, turn_index):
        1. ask_mode_enabled()?  no -> return None                (zero LLM/reads when off)
        2. question non-empty (trim, length-cap input)?  no -> return None
        3. load session; assignment-linked + targets resolvable?  no -> return None  (raw-tutor/free-chat → no Ask)
        4. fetch a small transcript window for context (last ~4 turns) — read-only
        5. msgs = build_ask_prompt(...)            [pure]
        6. client.chat.completions.create(model=ASK_MODEL, reasoning_effort='high', response_format=json_object)
        7. answer = parse_ask_answer(json.loads(resp), surface=...)   [pure]; None -> return None
        8. re-read session; append {question, answer, kind, turn_index, generated_at, model} to analysis_state['ask_log']
           via update_practice_session_analysis_state(...)
        9. return serialize_ask_answer(answer)            // {answer, kind} to the client; the provenance
                                                          // (question/turn_index/generated_at/model) rides the ask_log entry only
        (steps 3–8 wrapped in try/except -> return None; logger.exception; never raises)
     -> {success: True, ask: {answer, kind} | null}
  -> AskPanel appends the Q + answer to its visible list (+ a persistent "scaffold — not graded" note)
```

Ask works identically on voice and text (it's a text interaction in the sidecar; on voice it does NOT touch the realtime channel). The learner reads/types it between turns — never required while actively speaking.

## 6. Frontend detail

- **`api/ask.ts`** — `postAsk(sessionId, question, turnIndex?): Promise<AskAnswer | null>` via the shared axios client; `AskAnswer` = `{ answer: string; kind: 'hint'|'translation'|'definition'|'clarification'|'phrase'|'refusal' }`. Returns `response.data.ask ?? null`. Fail-open in the caller.
- **`components/learning/AskPanel.tsx`** — props `{ sessionId, currentTurnIndex? }`. A text input + send button; on submit calls `postAsk`, appends `{question, answer, kind}` to a local list; renders the running Q&A with a persistent footnote "Quick help is a scaffold — it isn't graded and doesn't count as your answer." A `refusal`-kind answer renders the same (it's already a redirect). Empty/failed response → a soft "Couldn't help with that right now — try rephrasing." Fail-open: a failed Ask never throws into the workspace.
- **`components/learning/ConversationSidecar.tsx`** — props `{ chips, sessionId, askModeEnabled, currentTurnIndex? }`. Renders a **Feedback | Ask** toggle. Feedback mode → the existing `<FeedbackSidecar chips={chips} />` (unchanged). Ask mode → `<AskPanel sessionId currentTurnIndex />`. When `askModeEnabled` is false, the Ask toggle is hidden and only Feedback renders (so the component is byte-equivalent to today's FeedbackSidecar mount when Ask is off).
- **Mount** — in `AssignmentConversationView` (`AssignmentPracticeWorkspace.tsx` ~line 613) swap `<FeedbackSidecar chips={coachChips} />` for `<ConversationSidecar chips={coachChips} sessionId={…} askModeEnabled={…} currentTurnIndex={…} />`. `askModeEnabled` comes from the launch bootstrap; `sessionId` from the active practice session.
- **Tests (vitest):** `postAsk` wrapper; `AskPanel` (submits → renders Q&A + scaffold note; refusal renders; failed/empty → soft message; fail-open); `ConversationSidecar` (toggle switches modes; Ask tab hidden when `askModeEnabled` false → renders Feedback only, equivalent to today).

## 7. Fail-open invariants

Every failure path → `ask: null` / a soft UI message — never a 500, never a blocked session: flag off · empty question · not assignment-linked · no targets · OpenAI error/timeout · malformed JSON (`parse_ask_answer` raises → caught). Flag off ⇒ the route gate skips bootstrap and the service gate precedes reads/LLM (only the cheap ownership lookup runs). The Ask tab is hidden when off (bootstrap `askModeEnabled=false`), so the learner isn't offered a dead affordance.

## 8. Flag & rollout — strangler-fig (same cadence as S1/S2/S3.1/S3.2/S3.3)

New flag **`PEDAGOGY_ENGINE_ASK_MODE`** (default off), independent of the other pedagogy flags:
1. Land behind the flag, default `'0'`; wire into `cloudbuild.yaml` `--set-env-vars` + `_PEDAGOGY_ENGINE_ASK_MODE: '0'` substitution. **REPLACE-safe** — brand-new flag, absent in live, so `'0'` matches. Verify all other substitution defaults still match live before any build.
2. Deploy inert (flag `'0'`) → verify health (Ask tab hidden; endpoint returns null doing minimal work).
3. Flip live: `--update-env-vars PEDAGOGY_ENGINE_ASK_MODE=1`.
4. Burn-in (text end-to-end: open Ask, ask "how do I say X?" → terse scaffolded answer; ask "just give me the answer" → a hint/redirect with `kind:"refusal"`; confirm the exchange lands in `ask_log` and NOT in `learning_events`) → bump cloudbuild default `'0'→'1'`.
5. **Rollback:** instant via `--update-env-vars PEDAGOGY_ENGINE_ASK_MODE=0`.

S3.4 has **no cutover dependency on S3.3** (independent flag, independent channel) — it can be cut over on its own merits whenever burn-in passes, regardless of the deferred S3.3 cutover.

## 9. Testing & eval

**Deterministic units (gate `make test-backend`)** — `backend/tests/test_pedagogy_engine_s3.py`:
- `parse_ask_answer`: coercion, length cap, kind normalization (unknown→clarification), empty-answer→None, non-dict→raises.
- `build_ask_prompt`: anti-answer-dump instruction present, scope/targets present, `feedback_policy.mode` + `scaffold_policy` threaded, voice terseness, `ui_language` threaded, strict-JSON instruction.
- `answer_ask` (fake db/client à la S2/S3.x): flag-off no-read/LLM; fail-open (raising db/client → None); assignment-only gate; no-targets → None; empty-question → None; `ask_log` append via `update_practice_session_analysis_state`; **`ask_log` write never touches `learning_events`** (assert the fake db's event path is untouched).
- **Extended `ImportBoundaryTestCase`** asserts `ask.py` imports no OpenAI/Canvas/resolver/compliance.
- Route tests (`test_curriculum_admin_*`): owner 404/403 first; flag-off → `ask:null` without bootstrap; flag-on → answer; fail-open 200/null on an unexpected error.

**Behavioral eval (opt-in `RUN_PEDAGOGY_EVAL=1`, NOT CI)** — seeded Ask questions (incl. explicit answer-dump attempts and out-of-scope requests) → real `gpt-5.4-mini` → LLM-judge on: does it refuse/redirect answer-dumps (the anti-outsourcing guardrail), stay in scope, stay terse, return responsibility. Reuses the S2/S3 eval scaffolding; model pinned.

**Live verify (playwright):** one assignment text session → open Ask → a scaffolded answer + an answer-dump redirect; confirm `ask_log` populated and `learning_events` unaffected; flag-off → Ask tab hidden.

## 10. As-built narrowing (to record in LIMITATIONS #53 after build)
- Ask is **text-only quick help**; replay + pronunciation comparison deferred.
- Ask answers are **scaffolds, not evidence** — excluded from `learning_events`/coverage/analysis by construction; the `ask_log` is learner-facing only (no L7 surface yet).
- The anti-answer-dump guardrail is **prompt-enforced + heuristic** (the `refusal` kind), not a hard classifier — its real fire rate needs the opt-in eval + production monitoring (like the other LLM guardrails).
- Single Q→A with a small context window; no multi-turn Ask memory.
