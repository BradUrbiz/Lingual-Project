# Pedagogy Engine S4.2 — Teacher Debrief (L7) — Design

**Status:** Design / approved by controller (autonomous build per the standing directive). Next: writing-plans.
**Date:** 2026-06-24
**Slice:** S4.2 (second sub-slice of §14 **S4 — Affect + debrief**; S4.1 affect shipped @ `b58a3d5`).
**Flag:** `PEDAGOGY_ENGINE_DEBRIEF` (default `'0'`, built-behind-flag, NOT cut over).

---

## 0. TL;DR

The Conversation Sidecar (S3) is *live, learner-facing*. **L7 is the after-conversation, teacher-facing counterpart**: a per-session **debrief** that packages the evidence a practice session already produced — hard-target coverage, learner self-repair, repeated-error families, the coach review, sidecar help *usage*, the affect readiness read — plus an honest statement of **what was not measured**, and a **suggested next practice**.

It is a **read-only presenter** over data that already exists in `practice_sessions` (`analysis_state` + `session_summary`). **No new persistence, no new compute pipeline, and NO new LLM call** — it assembles existing evidence; it does not mint new claims. This is the lightest engine slice: a pure presenter + one flag-gated teacher endpoint + one teacher page.

Two non-negotiables from the architecture: (1) **honesty over false precision** — the debrief states plainly that it is heuristic/transcript-based and *not* graded scoring, and ASR/pronunciation was not separately assessed (LIMITATIONS #7/#8); (2) **help ≠ evidence** — Ask usage is shown as counts/kinds only, never as student production.

---

## 1. Scope

### In scope
1. Pure `backend/services/pedagogy/debrief.py`: `build_session_debrief(session_record) -> dict` — assembles the teacher-facing debrief from an already-fetched session record (stdlib-only, import-boundary clean, no DB/OpenAI).
2. A read-only flag-gated teacher endpoint `GET /api/teacher/practice-sessions/<session_id>/debrief` — auth via the existing session→assignment→class→teacher chain; generate-on-read (no store); fail-soft.
3. `debrief_enabled()` gate in `integration.py` + flag `PEDAGOGY_ENGINE_DEBRIEF` (default `'0'`, REPLACE-safe cloudbuild wiring).
4. Frontend: `getSessionDebrief(sessionId)` in `frontend/src/api/teacher.ts` + a teacher `SessionDebriefPage` (or panel) + a click-through from the existing `recentSessions` list (gated on the flag via a launch/config signal).
5. Tests (pure presenter table, route auth + flag-off, frontend render) + doc-sync.

### Non-goals (deferred / out)
- **Per-claim ASR/pronunciation confidence caveats** — ASR confidence is NOT captured anywhere today (no field in the Realtime event, `useRealtimeChat`, event payload, `session_summary`, or `analysis_state`). The debrief uses a **static honesty disclaimer** instead. A future per-claim confidence surface needs an additive ASR-confidence capture first (documented follow-up).
- **No new LLM-generated narrative** (Approach B rejected). The debrief surfaces the *existing* S3.1 coach_review (itself a model pass); it does not generate new prose.
- **No assignment/class-level debrief roll-up** — per-session only (the arch's "after-conversation" granularity). An aggregate roll-up is a possible S4.2b. The existing assignment/class analytics already cover aggregates.
- **No teacher write-back / annotation** on the debrief (read-only). Teacher override of compiler inferences is L8, separate.
- **No new analysis_state key, no new store, no mutation** of the session.

## 2. Architecture — pure / impure split (invariant 7a)

```
PURE   backend/services/pedagogy/debrief.py            (stdlib only — import-boundary clean)
         • build_session_debrief(session_record: dict) -> dict
             Reads session_record['analysis_state'] + ['session_summary'] (+ a few top-level
             fields: status, started_at, ended_at, modality flags). Returns a JSON-able
             teacher-facing debrief dict (see §3). Pure, total, no-raise on malformed input
             (missing/!dict sub-objects degrade to empty sections + the disclaimer).

GATE   backend/services/pedagogy/integration.py
         • debrief_enabled()  (reads PEDAGOGY_ENGINE_DEBRIEF; mirrors the other flag helpers)

ROUTE  backend/routes/curriculum_admin.py
         • GET /api/teacher/practice-sessions/<session_id>/debrief
             flag gate → get_practice_session → _require_assignment_teacher_access(session.assignment_id)
             → build_session_debrief(session) → 200 {success, debrief}. Fail-soft (see §7).

FRONT  frontend/src/api/teacher.ts            • getSessionDebrief(sessionId) -> SessionDebrief
       frontend/src/pages/TeacherSessionDebriefPage.tsx   (new — renders the debrief sections)
       (+ a click-through from the recentSessions list, gated on the debrief flag)
```

**Why a pure presenter + dedicated endpoint (Approach A):** the debrief is a *projection* of existing evidence; keeping the assembly pure (stdlib, no DB) makes it fully unit-testable without a DB and keeps the import boundary clean, exactly like the other `pedagogy/*.py` modules. A dedicated per-session endpoint matches the arch's "after-conversation, per-session" framing and avoids bloating the aggregate analytics payloads.

## 3. Data contract — the debrief view

`build_session_debrief(session_record)` returns:

```jsonc
{
  "sessionId":   "<id>",
  "status":      "completed" | "active" | "abandoned",
  "startedAt":   "<iso8601>" | null,
  "endedAt":     "<iso8601>" | null,

  // Coverage of teacher targets (from session_summary hit maps + analysis_state['coverage'] if present)
  "coverage": {
    "expressionHits":  {"<expression>": <int>, ...},
    "vocabularyHits":  {"<vocab>": <int>, ...},
    "uncovered":       ["<target not yet hit>", ...],   // from analysis_state['coverage'] when present, else []
    "recycle":         ["<emerging target>", ...]
  },

  // Learner uptake / self-repair (from session_summary)
  "uptake": {
    "selfCorrectionCount": <int>,
    "feedbackCounts":      {"recast": <int>, "elicitation": <int>, "reviewItem": <int>},
    "taskCompletionCount": <int>
  },

  // Repeated-error families (from session_summary.repeated_error_counts)
  "repeatedErrors": [{"label": "<family>", "count": <int>}, ...],

  // The S3.1 model-verified coach review, surfaced verbatim if present (analysis_state['coach_review'])
  "coachReview": {"wins": [...], "workOn": [...], "targetCoverage": {...}} | null,

  // Promote-back log (analysis_state['promotions']) — what the coach surfaced into the main conversation
  "promotions": [ ... ] ,   // shape passed through as stored; [] if none

  // Help usage — COUNTS/KINDS ONLY (help != evidence). Never the question/answer content.
  "helpUsage": {
    "askCount": <int>,
    "byKind":   {"hint": <int>, "translation": <int>, "definition": <int>, "clarification": <int>, "phrase": <int>, "refusal": <int>}
  },

  // Affect readiness (analysis_state['affect_state']) — context for the teacher, not a grade
  "affect": {"readiness": "settled"|"neutral"|"strained", "reason": "<short>"} | null,

  // Suggested next practice — DERIVED heuristically (no LLM): uncovered/recycle targets + top repeated errors (+ coachReview.workOn when present)
  "suggestedNext": ["<short actionable suggestion>", ...],

  // Honesty disclaimer — ALWAYS present (the L7 "honesty over false precision" contract)
  "caveats": [
    "This debrief summarizes the practice transcript. Target and error detection is heuristic, not graded scoring.",
    "Pronunciation and listening accuracy were not separately assessed.",
    "Help requests are shown as usage counts, not as evidence the learner produced the form."
  ]
}
```

All sub-sections degrade to empty (`[]`/`{}`/`null`) when the underlying data is missing; `caveats` is always present.

## 4. Suggested-next-practice heuristic (no LLM)

Deterministic, derived from existing evidence (order = priority):
1. `analysis_state['coverage'].uncovered` → "Revisit: <targets not yet used>."
2. top `repeated_error_counts` families (count desc, take ≤2) → "Focus on recurring issue: <family>."
3. `coach_review.work_on` (if present, ≤2) → surfaced as suggestions.
4. `analysis_state['coverage'].recycle` (emerging) → "Reinforce: <emerging targets>."

Cap the list at `MAX_SUGGESTIONS = 4`, de-duplicated, stable order. If no evidence yields any suggestion, return `[]` (the debrief still renders coverage + caveats).

## 5. Integration & flow

1. Teacher opens a session's debrief (click-through from the `recentSessions` list, which already carries `id`).
2. `GET /api/teacher/practice-sessions/<id>/debrief`: `debrief_enabled()` gate (flag-off → 404/`{success:false}` doing minimal work) → `get_practice_session(id)` (404 if missing) → `_require_assignment_teacher_access(deps, session['assignment_id'])` (403 if not the owning teacher) → `build_session_debrief(session)` → `200 {success:true, debrief}`.
3. Frontend renders the sections; the recentSessions click-through and the page link are shown only when the flag-derived config says debrief is enabled.

**Read-only:** no write to the session, no new `analysis_state` key, no LLM. Generate-on-read each time (cheap — pure dict assembly).

## 6. Flag & rollout

- `PEDAGOGY_ENGINE_DEBRIEF`, `debrief_enabled()` in `integration.py` (mirror `affect_enabled`), cloudbuild `_PEDAGOGY_ENGINE_DEBRIEF: '0'` + appended to the `--set-env-vars` REPLACE string. REPLACE-safe: brand-new flag absent live, default `'0'`; no other substitution default touched (RECYCLING/COACH_REVIEW/COACH_CHIPS=`'1'`, PROMOTE_BACK/ASK_MODE/AFFECT=`'0'`).
- Surfacing the flag to the frontend (decided): the existing assignment + student-drill-down analytics **routes** attach `debriefEnabled: debrief_enabled()` to their JSON payload (an additive boolean set in the route handler, NOT in the pure payload builders). The frontend renders the per-session debrief click-through on each `recentSessions` row **only when `debriefEnabled` is true** — no dead links, no new config endpoint. The debrief page itself also fail-softs (if reached with the flag off, the endpoint's `{success:false}` renders a "not available" state).
- Built behind flag, NOT cut over. Independent of all other flags. Cutover later after a teacher reviews the debrief framing on real sessions.

## 7. Error handling

- Flag-off → endpoint returns `{success:false}` (or 404) doing minimal work; the gate is before any DB read.
- Missing session → 404; not-owning-teacher → 403 (the existing `SchoolContextPermissionError`/`ValueError` → 403/404 pattern).
- `build_session_debrief` is total/no-raise: malformed/missing `analysis_state`/`session_summary` sub-objects degrade to empty sections; `caveats` always present. The route wraps the build in a try → on unexpected error returns a minimal debrief (`caveats` + `sessionId`/`status`) with `200`, never a 500. Read-only, so fail-soft (degrade the view) rather than fail-closed.

---

## 8. Testing

- **Pure presenter table** (`test_pedagogy_engine_s4.py`): full session record → all sections populated; missing `analysis_state` → empty sections + caveats; missing `coach_review` → `coachReview: null`; help-usage counts derived from `ask_log` (and asserts NO ask question/answer text leaks into the output); suggested-next from uncovered + repeated errors; caveats always present; malformed input → no raise.
- **Route** (`test_curriculum_admin_*`): flag-off → minimal-work `{success:false}`/404 (assert `get_practice_session` not called when off, or returns before auth); missing session → 404; non-owning teacher → 403; owner → 200 with debrief; build-error → fail-soft 200 minimal (not 500).
- **Import boundary**: `debrief.py` added to `ImportBoundaryTestCase`.
- **Frontend**: `getSessionDebrief` returns the debrief; the page renders sections + caveats; the recentSessions click-through is hidden when the flag is off.
- Deterministic units gate `make test-backend` / `npm run test`.

## 9. Invariants (the review lens)

1. **Read-only / no mutation** — no write to the session, no new `analysis_state` key, no new store.
2. **No new LLM** — pure dict assembly; the only model output shown is the *already-stored* S3.1 coach_review.
3. **Help ≠ evidence** — Ask usage is counts/kinds only; NO question/answer content in the debrief.
4. **Honesty over false precision** — `caveats` always present; the debrief never claims graded scoring or pronunciation assessment.
5. **Teacher-scoped auth** — session→assignment→class→teacher via `_require_assignment_teacher_access`; non-owners get 403, missing 404.
6. **Fail-soft** — never 500s; degrades to a minimal debrief; flag-off does minimal work before any read.
7. **Import boundary (7a)** — `debrief.py` stdlib-only.
8. **Additive** — no existing analytics endpoint/payload or student/tutor path changed.
