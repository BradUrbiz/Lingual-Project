# S4.2b — Assignment-Level Teacher Debrief Roll-Up (design)

**Date:** 2026-06-25
**Status:** design approved (self-directed under standing delegation); build behind flag, default off
**Flag:** `PEDAGOGY_ENGINE_DEBRIEF_ROLLUP` (new, default `0`); gate = `debrief_enabled() AND debrief_rollup_enabled()`
**Builds on:** S4.2 session debrief (`backend/services/pedagogy/debrief.py`, `build_session_debrief`)

---

## 1. Problem & purpose

S4.2 gives a teacher a **per-session** debrief: how one student's one practice
session went. There is no class-level view of *how the AI tutor coached across
the whole assignment*. A teacher running an assignment for 25 students has 25
session debriefs and no roll-up.

The existing **assignment analytics** page
(`GET /api/teacher/assignments/<id>/analytics`) already aggregates the
**content/coverage** dimension across an assignment: target-expression hits,
repeated errors, rubric scores, objectives, session/student counts, recent
sessions. We must **not** rebuild that.

The session debrief's *distinctive* contribution — data that lives only in
`practice_sessions.analysis_state` / `session_summary` and appears on no other
teacher screen — is the **engine's coaching process**: feedback uptake, S3.3
promote-back corrections, S5 director re-steers, Ask-mode help demand, and S4.1
affect. **S4.2b rolls up exactly those process signals across the assignment**,
and links to analytics for content coverage.

**One-line scope:** *Across this assignment's sessions, how did the AI tutor
coach — interventions, feedback uptake, help demand, and affect — so the teacher
sees the teaching process at class scale.*

## 2. Approaches considered

- **A — "Coaching Roll-Up" (CHOSEN).** Aggregate only the engine-process
  signals analytics does not show (uptake, promotions, re-steers, help, affect),
  plus a participation denominator; link to analytics for coverage. Sharp
  purpose, DRY, no dual source of truth for coverage.
- **B — "Full mirror."** Re-aggregate the whole session debrief including
  coverage and repeated errors. *Rejected:* duplicates assignment analytics →
  two screens that can disagree; more surface to keep consistent (YAGNI/DRY).
- **C — "Inline into analytics page."** Add coaching aggregates as a section on
  the always-on analytics endpoint. *Rejected:* couples a flag-gated engine
  feature to the always-on analytics payload, complicating the gate; the session
  debrief is its own page, so an assignment debrief page is the consistent model.
  We keep C's discoverability by adding a **link** from analytics → the debrief
  page, gated on the flag.

## 3. Architecture

A new **pure presenter**, mirroring S4.2's contract exactly:

```python
# backend/services/pedagogy/assignment_debrief.py  (stdlib-only, invariant 7a)
def build_assignment_debrief(session_records: list) -> dict:
    """Aggregate practice_session records into an assignment-level coaching
    roll-up. Pure, stdlib-only, no DB/LLM/store. Calls build_session_debrief()
    per record for the teacher-safe coaching shapes; reads raw status /
    student_firebase_uid / started_at for the participation denominator.
    Total / no-raise: malformed records degrade to empty sections; ``caveats``
    is always present."""
```

- **Why take raw records, not pre-built debriefs:** the per-session debrief
  output intentionally omits `student_firebase_uid`, so the participation
  denominator (distinct student count) can only come from the raw record. The
  aggregator calls `build_session_debrief(r)` internally for the teacher-safe
  coaching shapes (DRY — single source of per-session shaping) and reads raw
  `status` / `student_firebase_uid` / `started_at` for participation. It remains
  pure (input list → output dict; no DB/LLM), so it is unit-testable with fake
  records.
- **No new persistence, no new analytics write.** Read path only; reuses
  `deps.db.list_assignment_practice_sessions(assignment_id)` which the analytics
  route already calls (PG-authoritative; `WRITE_FIRESTORE_ANALYTICS=0`
  unaffected).

### 3.1 Return shape

```python
{
  "assignmentId": str | None,          # from first record's assignment_id
  "participation": {
    "sessionCount": int,               # len(records)
    "completedSessionCount": int,      # status == "completed"
    "studentCount": int,               # distinct student_firebase_uid (non-empty)
    "firstStartedAt": <value> | None,  # min started_at, defensive (see §3.2)
    "lastStartedAt": <value> | None,   # max started_at, defensive
  },
  "uptake": {                          # summed across all sessions
    "selfCorrectionCount": int,
    "feedbackCounts": {"recast": int, "elicitation": int, "reviewItem": int},
    "taskCompletionCount": int,
  },
  "promotions": {                      # S3.3 targeted corrections, pooled by target
    "count": int,                      # total promotion events across sessions
    "byTarget": [                      # sorted by count desc, then target asc; top 10
      {"target": str, "count": int, "sessionCount": int}, ...
    ],
  },
  "directorReSteers": {                # S5 interventions, pooled
    "count": int,
    "byKind": {kind: int},             # only kinds that occurred
    "byTarget": [{"target": str, "count": int}, ...],  # sorted, top 10; empty targets dropped
  },
  "helpUsage": {                       # Ask-mode demand, summed
    "askCount": int,
    "byKind": {kind: int},             # the six _ASK_KINDS, summed
    "sessionsWithHelp": int,           # sessions with askCount > 0
  },
  "affect": {                          # S4.1 readiness distribution
    "byReadiness": {readiness: int},   # session count per readiness label
    "sessionsWithSignal": int,         # sessions where affect was present
  },
  "coachReview": {"sessionCount": int},# sessions that had a coach_review present
  "suggestedNext": [str, ...],         # synthesized from aggregates, max 4
  "caveats": [...],                    # S4.2 caveats + one assignment-scoped caveat
}
```

**Deliberately absent** (owned by assignment analytics, linked instead):
content coverage, target-expression hits, repeated-error list, rubric scores,
objectives. The roll-up is the coaching-process view, not a second analytics
screen.

### 3.2 Aggregation rules (deterministic, pure)

- **participation.studentCount:** number of distinct non-empty
  `student_firebase_uid` across records.
- **participation.first/lastStartedAt:** `min`/`max` over non-None `started_at`
  values. **Defensive:** wrap in `try/except TypeError` (mixed/uncomparable
  types) → `None` on failure. Do not assume input ordering.
- **uptake.\*:** element-wise sum of each session debrief's `uptake` numbers.
- **promotions:** `count` = sum of each session's `promotions.count`. `byTarget`
  groups all `promotions.items[].target` (non-empty) → `{target, count,
  sessionCount}` where `count` = total events for that target, `sessionCount` =
  number of distinct sessions that promoted it. Sort by `count` desc, then
  `target` asc; cap at top 10.
- **directorReSteers:** `count` = sum. `byKind` counts non-empty `kind` values.
  `byTarget` groups non-empty `target` values → `{target, count}`, sort + cap as
  above.
- **helpUsage:** `askCount` = sum of session `helpUsage.askCount`; `byKind` sums
  each of the six `_ASK_KINDS`; `sessionsWithHelp` = sessions with `askCount>0`.
- **affect:** for each session whose debrief `affect` is non-None, increment
  `byReadiness[readiness]` (skip None/empty readiness) and `sessionsWithSignal`.
- **coachReview.sessionCount:** sessions whose debrief `coachReview` is non-None.
- **suggestedNext** (heuristic, deterministic, max 4, de-duped, stable order):
  1. Top `promotions.byTarget` entry with `sessionCount >= 2`: `"Multiple
     students needed correction on {target} — consider a focused mini-lesson."`
  2. If `affect.byReadiness["strained"] >= 2` (S4.1 readiness labels are
     exactly `settled | neutral | strained`): `"{n} sessions showed signs of
     strain — consider easing pace or difficulty."`
  3. If `helpUsage.askCount >= 2 * max(participation.sessionCount, 1)` (heavy
     help demand): `"Students leaned on help often — the task may be above their
     current level."`
  4. If none of the above fired and `participation.completedSessionCount > 0`:
     `"The class handled this assignment's targets well — consider advancing
     difficulty."`
- **caveats:** the three S4.2 `_CAVEATS` plus: `"This roll-up aggregates {N}
  session(s) across {M} student(s); per-student detail is in each session
  debrief."` (N/M filled from participation).

### 3.3 Empty / malformed inputs

`build_assignment_debrief([])` → all counts 0, all lists empty, all dicts
present, `assignmentId=None`, `caveats` present (N=0, M=0). Each record is run
through `build_session_debrief` (already total/no-raise); raw-field reads use
`.get` with guards. The function never raises.

## 4. Backend route

```python
# backend/routes/curriculum_admin.py  (next to the session-debrief route)
@bp.route('/api/teacher/assignments/<assignment_id>/debrief', methods=['GET'])
@deps.login_required
def api_get_assignment_debrief(assignment_id):
    if not (debrief_enabled() and debrief_rollup_enabled()):
        return jsonify({'success': False, 'error': 'Assignment debrief is not enabled.'}), 200
    _require_assignment_teacher_access(deps, assignment_id)   # 404/403 as elsewhere
    try:
        sessions = deps.db.list_assignment_practice_sessions(assignment_id)
        debrief = build_assignment_debrief(sessions)
    except Exception:
        debrief = build_assignment_debrief([])   # fail-soft: minimal, caveats present
    return jsonify({'success': True, 'debrief': debrief})
```

- **Auth:** identical to the session-debrief and analytics routes —
  `@deps.login_required` + `_require_assignment_teacher_access`. No new auth
  surface.
- **Gate:** both flags. With `PEDAGOGY_ENGINE_DEBRIEF=1` already LIVE in prod,
  activation in practice requires only flipping `PEDAGOGY_ENGINE_DEBRIEF_ROLLUP`.
- **Flag helper** in `backend/services/pedagogy/integration.py`, mirroring
  `debrief_enabled()`:
  ```python
  def debrief_rollup_enabled() -> bool:
      """S4.2b assignment-level roll-up (independent flag, default off).
      Reads PEDAGOGY_ENGINE_DEBRIEF_ROLLUP. Meaningful only with
      debrief_enabled() also on (the route requires both)."""
      return os.environ.get("PEDAGOGY_ENGINE_DEBRIEF_ROLLUP", "").strip().lower() in _TRUTHY
  ```
- **Analytics response gains a flag:** add `'debriefRollupEnabled':
  debrief_rollup_enabled()` to the existing
  `GET /api/teacher/assignments/<id>/analytics` JSON so the frontend can show the
  link. (Additive key; off → `false` → link hidden → byte-equivalent UI.)

## 5. Frontend

- **Types** (`frontend/src/api/teacher.ts`): new `AssignmentDebrief` interface
  matching §3.1; `getAssignmentDebrief(assignmentId)` API fn calling
  `GET /teacher/assignments/{id}/debrief` (same `{success, debrief}` envelope as
  the session debrief). Add optional `debriefRollupEnabled?: boolean` to the
  assignment-analytics response type.
- **Page** (`frontend/src/pages/TeacherAssignmentDebriefPage.tsx`): mirrors
  `TeacherSessionDebriefPage` structure, assignment-scoped. Cards: Participation,
  Feedback uptake, Targeted corrections (promotions.byTarget), Coaching
  interventions (directorReSteers), Help usage, Affect, Coach reviews,
  Suggested next, Caveats. Each card **self-hides when its section is empty /
  zero**, mirroring the session page (so a roll-up over sessions with the engine
  intervention flags off renders only Participation + Caveats — honest, not
  misleading). Route: `/app/teacher/assignments/:assignmentId/debrief`
  (lazy-loaded in `App.tsx`).
- **Link** in `TeacherAssignmentAnalyticsPage.tsx`: a "View coaching debrief"
  link to the new page, rendered only when `debriefEnabled &&
  debriefRollupEnabled` (both come from the analytics response). Off → no link →
  page UI byte-equivalent to today.

## 6. Flag wiring & docs

- `cloudbuild.yaml`: add `_PEDAGOGY_ENGINE_DEBRIEF_ROLLUP: '0'` substitution
  default and `PEDAGOGY_ENGINE_DEBRIEF_ROLLUP=${_PEDAGOGY_ENGINE_DEBRIEF_ROLLUP}`
  to the `--set-env-vars` line. **REPLACE-safety:** this only *adds* a var at
  `'0'`; audit the full set-env-vars/set-secrets coverage before any actual
  build (no live var dropped, no value changed). Build-behind-flag ≠ cutover.
- Docs: `backend/CLAUDE.md` flag-state row (BUILT, default off, NOT cut over);
  `docs/school-integration/LIMITATIONS.md` entry; `docs/school-integration/
  TASKS.md` item; the pedagogy engine doc (S4.2 section) gets an S4.2b note.

## 7. Testing

- **Backend unit** (`backend/tests/test_pedagogy_engine_s4.py`, new
  `BuildAssignmentDebriefTestCase`): empty list; single session; multi-session
  pooling (uptake sums, promotions byTarget grouping + sessionCount, resteer
  byKind/byTarget, helpUsage sums + sessionsWithHelp, affect distribution,
  coachReview count); malformed records degrade not raise; suggestedNext each
  branch; caveats always present with N/M; first/last started-at defensive on
  uncomparable types; `debrief_rollup_enabled()` flag reading.
- **Backend route** (`backend/tests/test_curriculum_admin_coach_chip_route.py`
  or sibling): both-flags-on returns debrief; rollup-flag-off returns
  `{success: false}` with minimal work; non-teacher → 403; unknown assignment →
  404; build error → fail-soft minimal debrief (not 500); analytics response
  carries `debriefRollupEnabled`.
- **Frontend** (`TeacherAssignmentDebriefPage.test.tsx`): renders all cards with
  full data; empty roll-up renders only Participation + Caveats (others
  self-hide); link in analytics page shows only when both flags true.

## 8. Non-goals (v1)

- No content-coverage / rubric / expression-hit aggregation (analytics owns it).
- No per-student breakdown table (session debriefs already give per-session
  detail; the roll-up is class-level).
- No LLM-synthesized narrative; `suggestedNext` stays deterministic/heuristic.
- No CSV/export, no time-series trend across assignments.
- No coach-review content rollup (opaque shape) — only a present-count.

## 9. Invariants preserved

- Pure presenter: no LLM, no store, no new persistence; stdlib-only import
  boundary (invariant 7a).
- HELP ≠ EVIDENCE: help usage shown as counts, caveat carried over.
- Honesty caveats always present; self-hiding cards prevent implying signal
  that is not there (e.g., engine intervention flags off).
- Byte-equivalent when flag off (route returns `{success:false}`; no link).
- Fail-soft: any error → minimal debrief, never a 500.
